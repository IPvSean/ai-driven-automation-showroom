== Background: Agentic Automation

Traditional automation follows a predefined path: an event triggers a workflow, the workflow runs a sequence of job templates, and the result is logged. The human designs the workflow ahead of time.

**Agentic automation** adds an AI layer that can:

* **Discover** — query systems to understand the current state
* **Reason** — determine what actions are needed based on what it finds
* **Propose** — suggest specific remediation steps to the operator
* **Execute** — carry out approved actions through existing automation

The key difference is that the agent can handle *novel situations* without a pre-built workflow. It uses the tools available to it (AAP inventory, job templates, Insights data) to figure out what to do and then asks for human approval.

[quote]
💡 **Think of it this way**: Traditional automation is like a train on rails — fast and reliable, but it only goes where tracks exist. Agentic automation is like a GPS-guided driver — it can navigate new routes, but you're sitting in the passenger seat and can say "stop" at any time.

== Architecture Overview

The agentic IDE workflow connects three components:

[source]
----
Developer IDE (with AI Agent)
    ↕ MCP Protocol
Ansible MCP Server
    ↕ REST API
Ansible Automation Platform
    (Inventory, Job Templates, Credentials, Jobs)
----

* **Developer IDE** — The coding environment with an integrated AI chat agent
* **Ansible MCP Server** — A Model Context Protocol server that exposes AAP capabilities (inventory, jobs, templates) as tools the AI agent can call
* **Ansible Automation Platform** — The execution engine that manages inventory, credentials, and runs automation

[quote]
💡 **What is MCP?** The Model Context Protocol is an open standard that allows AI assistants to interact with external tools and data sources. The Ansible MCP server exposes AAP's REST API as MCP tools, so AI agents can query inventory, list job templates, launch jobs, and more — all through natural language conversation.

== The Ansible MCP Server

The MCP server provides the AI agent with the following capabilities:

[cols="2,3", options="header"]
|===
| MCP Tool Category | What the Agent Can Do

| **Inventory**
| List hosts, groups, and inventories; inspect host variables; check which systems are registered

| **Job Templates**
| List available job templates; read template configurations; launch jobs with parameters

| **Jobs**
| Check running and completed jobs; read job output and events; monitor job status

| **Projects**
| List and sync projects from Git repositories

| **Credentials**
| List credential types (not secrets) to understand what access is available
|===

The agent calls these tools behind the scenes when you ask questions in natural language. You don't need to know the API endpoints — just describe what you want in plain English.

== Agentic Workflow: Server Assessment and Patching

In this exercise, you will have a conversation with the AI agent to assess your RHEL fleet and remediate issues discovered in Section 7.

=== Step 1: Ask the agent about your servers

In the AI Chat panel, type a prompt like:

[source]
----
Are there any servers in my inventory that need patching or have outstanding
remediation actions?
----

Observe how the agent:

* Queries the AAP inventory via MCP to list your RHEL hosts
* Checks for available job templates related to patching and remediation
* Reports back with a summary of what it found

== Extended Exercise: Multi-Host Remediation

Try a broader prompt to see the agent handle multiple hosts:

[source]
----
Check all three RHEL servers. Are there any outstanding issues? Can you
remediate everything that has a matching job template?
----

The agent should:

1. Query all RHEL hosts in the inventory
2. Cross-reference available job templates
3. Present a remediation plan for your approval
4. Execute approved actions one at a time (or propose a workflow)

[quote]
💡 **This is the power of agentic automation.** Instead of navigating multiple AAP screens, Insights dashboards, and Git repositories, you had a single conversation that assessed your fleet and executed remediation. The agent did the research; you made the decisions.

== How This Differs from Traditional Workflows

[cols="2,2,2", options="header"]
|===
| Aspect | Traditional (Parts 1-3) | Agentic (This Section)

| **Trigger**
| Event (Filebeat, Splunk, Winlogbeat)
| Human prompt in IDE

| **Decision Maker**
| Pre-defined workflow logic
| AI agent with human approval

| **Flexibility**
| Fixed workflow steps
| Dynamic — agent adapts to what it finds

| **Approval**
| Built into workflow (optional)
| Always required — agent asks before acting

| **Best For**
| Known, repeatable scenarios
| Ad-hoc investigation and remediation
|===

== Key Takeaways

* **Agentic automation** lets you use natural language to manage infrastructure through an AI assistant
* The **Ansible MCP server** bridges the AI agent to Ansible Automation Platform's full capabilities
* **Human-in-the-loop** means the agent proposes but you decide — every action requires explicit approval
* The agent can **discover, reason, and propose** actions across inventory, job templates, and job history
* This approach complements traditional event-driven workflows — use both depending on the situation
