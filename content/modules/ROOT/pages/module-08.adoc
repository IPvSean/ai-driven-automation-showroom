== Background: Agentic Automation

Traditional automation follows a predefined path: an event triggers a workflow, the workflow runs a sequence of job templates, and the result is logged. The human designs the workflow ahead of time.

**Agentic automation** adds an AI layer that can:

* **Discover** — query systems to understand the current state
* **Reason** — determine what actions are needed based on what it finds
* **Propose** — suggest specific remediation steps to the operator
* **Execute** — carry out approved actions through existing automation

The key difference is that the agent can handle *novel situations* without a pre-built workflow. It uses the tools available to it (AAP inventory, job templates, Red Hat Lightspeed data) to figure out what to do and then asks for human approval.

[quote]
💡 **Think of it this way**: Traditional automation is like a train on rails — fast and reliable, but it only goes where tracks exist. Agentic automation is like a GPS-guided driver — it can navigate new routes, but you're sitting in the passenger seat and can say "stop" at any time.

== Architecture Overview

The agentic IDE workflow connects three components:

[source]
----
code-server (VS Code in browser)
    └── Claude Code (AI agent in terminal)
            ↕ MCP Protocol
        Ansible MCP Server
            ↕ REST API
        Ansible Automation Platform
            (Inventory, Job Templates, Credentials, Jobs)
----

* **code-server** — Visual Studio Code running in your browser, providing a full IDE experience
* **Claude Code** — An AI agent that runs in the terminal (`claude` command) and can call external tools via MCP
* **Ansible MCP Server** — Exposes AAP capabilities (inventory, jobs, templates) as MCP tools that Claude Code can call
* **Ansible Automation Platform** — The execution engine that manages inventory, credentials, and runs automation

[quote]
💡 **What is MCP?** The Model Context Protocol is an open standard that allows AI assistants to interact with external tools and data sources. The Ansible MCP server exposes AAP's REST API as MCP tools, so Claude Code can query inventory, list job templates, launch jobs, and more — all through natural language conversation.

== The Ansible MCP Server

The MCP server provides Claude Code with the following capabilities:

[cols="2,3", options="header"]
|===
| MCP Tool Category | What Claude Code Can Do

| **Inventory**
| List hosts, groups, and inventories; inspect host variables; check which systems are registered

| **Job Templates**
| List available job templates; read template configurations; launch jobs with parameters

| **Jobs**
| Check running and completed jobs; read job output and events; monitor job status

| **Projects**
| List and sync projects from Git repositories

| **Credentials**
| List credential types (not secrets) to understand what access is available
|===

Claude Code calls these tools behind the scenes when you ask questions in natural language. You don't need to know the API endpoints — just describe what you want in plain English.

== How This Differs from Traditional Workflows

[cols="2,2,2", options="header"]
|===
| Aspect | Traditional (Parts 1-3) | Agentic (This Section)

| **Trigger**
| Event (Filebeat, Splunk, Winlogbeat)
| Human prompt in Claude Code

| **Decision Maker**
| Pre-defined workflow logic
| AI agent with human approval

| **Flexibility**
| Fixed workflow steps
| Dynamic — Claude Code adapts to what it finds

| **Approval**
| Built into workflow (optional)
| Always required — Claude Code asks before acting

| **Best For**
| Known, repeatable scenarios
| Ad-hoc investigation and remediation
|===

== Key Takeaways

* **Claude Code and OpenCode** are terminal-based AI agents that run inside code-server and connect to AAP via MCP — use whichever you prefer
* The **Ansible MCP server** bridges any MCP-compatible AI agent to Ansible Automation Platform's full capabilities
* **Human-in-the-loop** means the agent proposes but you decide — every action requires explicit approval
* The agent can **discover, reason, and propose** actions across inventory, job templates, and job history
* This approach complements traditional event-driven workflows — use both depending on the situation
